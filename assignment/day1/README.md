# 여행의 발견

국내 여섯 도시의 여행 정보를 살펴보고 나만의 여행 일정과 준비 항목을 관리할 수 있는 웹사이트입니다. 여러 페이지를 목적에 따라 분리하고, JavaScript ES Modules와 Local Storage를 활용해 사용자별 데이터를 유지하도록 구현했습니다.

## 실행 결과

### 메인 페이지

![여행의 발견 메인 페이지](docs/images/main.png)

- 서비스의 콘셉트와 국내 추천 여행지를 소개합니다.
- 서울·광주·울산 배경이 순환하고, 영상 캐러셀과 오늘의 여행 문구를 제공합니다.
- JavaScript 데이터로 여섯 도시의 여행지 카드를 동적으로 생성합니다.

### 여행지 상세 페이지

![여행지 상세 페이지](docs/images/destination.png)

- URL의 `city` 쿼리 파라미터에 따라 도시별 내용을 동적으로 표시합니다.
- 도시 소개, 추천 명소, 대표 먹거리, 여행 팁과 영상을 제공합니다.
- 해당 도시가 선택된 상태로 여행 계획 페이지에 이동할 수 있습니다.

### 여행 계획 페이지

![여행 계획 페이지](docs/images/planner.png)

- 여행지와 기간을 선택해 여행을 생성하고 수정·삭제할 수 있습니다.
- 날짜별 시간 일정과 준비 항목을 추가하고 완료 상태를 관리합니다.
- 일정 필터, 달력, 여행 기간과 D-day를 제공합니다.
- 로그인한 사용자별 여행 계획을 Local Storage에 저장합니다.

### 회원가입 페이지

![회원가입 페이지](docs/images/signup.png)

- 사용자 이름, 아이디, 비밀번호와 추가 프로필 정보를 입력받습니다.
- 중복 아이디와 입력값을 검사하고 비밀번호를 PBKDF2 방식으로 처리합니다.
- 로그인한 사용자에 따라 여행 계획 데이터를 구분합니다.

### 로그인 페이지

![로그인 페이지](docs/images/login.png)

- 저장된 회원 아이디와 비밀번호를 검증해 로그인 세션을 생성합니다.
- 로그인하지 않은 상태에서 플래너에 접근하면 로그인 후 원래 페이지로 돌아갈 수 있도록 이동 경로를 전달합니다.
- 로그인 후에는 내비게이션의 로그인·회원가입 링크가 사용자 이름과 로그아웃 버튼으로 변경됩니다.

### 회원 목록 페이지

![회원 목록 페이지](docs/images/result.png)

- 현재 브라우저의 Local Storage에 등록된 실습용 회원 정보를 표로 보여줍니다.
- JavaScript가 회원 데이터를 읽어 `<tbody>`에 행을 동적으로 생성합니다.
- 화면이 좁을 때는 표 영역을 가로로 스크롤해 내용을 확인할 수 있습니다.

| 다크모드 | 모바일 화면 |
| --- | --- |

- CSS 변수 값을 변경하는 방식으로 라이트·다크 테마를 전환합니다.
- 선택한 테마는 Local Storage에 저장하고, 처음 방문하면 운영체제 설정을 참고합니다.
- 모바일에서는 내비게이션을 햄버거 메뉴로 변경하고 카드와 폼을 한 열로 재배치합니다.

## 기술 선택과 설계 이유

### `<head>`와 `<header>`의 역할

이름은 비슷하지만 두 요소의 목적은 다릅니다.

| 요소 | 역할 | 화면 표시 | 이 프로젝트에서의 사용 |
| --- | --- | --- | --- |
| `<head>` | 문서 제목, 설명, 문자 인코딩과 스타일시트 연결 정보 | 직접 표시되지 않음 | 페이지별 메타 정보와 공통 CSS 연결 |
| `<header>` | 로고와 내비게이션 같은 화면의 머리말 | 표시됨 | 모든 페이지의 브랜드와 주요 메뉴 구성 |

JavaScript 모듈은 HTML 해석을 방해하지 않도록 각 문서의 `body` 끝에서 연결했습니다. 화면에 보이는 `<header>`에는 다음과 같이 `position: sticky`를 적용해 스크롤 후에도 메뉴를 사용할 수 있게 했습니다.

```css
.site-header {
  position: sticky;
  top: 0;
  z-index: 100;
}
```

### Sticky와 Fixed

| 구분 | Sticky | Fixed |
| --- | --- | --- |
| 기준 | 원래 문서 위치에서 시작해 지정 위치에 도달하면 고정 | 처음부터 브라우저 화면을 기준으로 고정 |
| 문서 흐름 | 유지 | 제외 |
| 부모 영역 | 부모 영역 안에서 동작 | 일반적으로 부모 범위와 관계없이 뷰포트 기준 |
| 적합한 요소 | 헤더, 사이드바, 목차 | 플로팅 버튼, 채팅 버튼, 화면 고정 도구 |
| 프로젝트 적용 | 상단 헤더, 회원가입 안내, 여행 목록 | 맨 위로 이동 버튼, 배경 장식 |

상단 메뉴는 처음부터 콘텐츠를 덮는 Fixed보다 문서 흐름을 유지하는 Sticky가 적합하다고 판단했습니다. 반면 맨 위로 버튼은 어느 영역에서든 같은 위치에 있어야 하므로 Fixed를 선택했습니다.

Sticky의 장점은 필요한 메뉴와 여행 목록을 스크롤 중에도 확인할 수 있다는 점입니다. 다만 부모 요소의 `overflow`나 높이에 영향을 받을 수 있고 작은 화면에서는 콘텐츠 공간을 차지할 수 있습니다. 따라서 `900px` 이하에서는 회원가입 안내와 플래너 사이드바를 `position: static`으로 되돌렸습니다.

### Grid와 Flexbox

| 구분 | CSS Grid | Flexbox |
| --- | --- | --- |
| 배치 방향 | 행과 열을 함께 관리하는 2차원 배치 | 한 방향을 중심으로 하는 1차원 배치 |
| 적합한 UI | 카드 목록, 폼, 달력, 전체 페이지 레이아웃 | 내비게이션, 버튼, 한 줄 정렬 |
| 프로젝트 적용 | 여행지 카드, 명소, 폼, 플래너, 7열 달력 | 메뉴, 버튼 그룹, 일정 항목, 푸터 |

여행지와 명소 카드는 같은 크기의 열이 반복되므로 Grid를 사용했습니다. 화면 너비에 따라 3열, 2열, 1열로 바꾸기 쉽다는 장점이 있습니다. 내비게이션과 일정 항목은 한 방향으로 정렬하고 요소 사이 간격을 조절하는 것이 중요하므로 Flexbox를 사용했습니다.

Grid는 복잡한 행·열 배치를 명확하게 표현할 수 있지만 단순한 한 줄 정렬에는 코드가 불필요하게 커질 수 있습니다. Flexbox는 한 방향 정렬에는 간결하지만 여러 행과 열을 동시에 맞추는 레이아웃에서는 너비 계산과 줄바꿈 관리가 복잡해질 수 있습니다.

### 시맨틱 태그와 `div`

모든 영역을 `div`로 작성할 수도 있지만 콘텐츠의 의미를 명확하게 전달하기 위해 시맨틱 태그를 선택했습니다.

| 태그 | 적용 내용 | 선택 이유 |
| --- | --- | --- |
| `header`, `nav` | 브랜드와 주요 메뉴 | 페이지 머리말과 이동 영역을 명확히 표현 |
| `main` | 페이지의 핵심 콘텐츠 | 문서의 주요 영역을 보조 기술에 전달 |
| `section` | 여행지, 영상, 플래너 영역 | 주제별 콘텐츠 그룹 구분 |
| `article` | 독립적으로 이해할 수 있는 음식 콘텐츠 | 자체 제목과 내용을 가진 콘텐츠 표현 |
| `aside` | 회원가입 안내, 여행 목록, 음식 팁 | 본문을 보조하는 정보 표현 |
| `figure`, `figcaption` | 이미지·영상과 설명 | 미디어와 설명의 관계 표현 |
| `footer` | 저작권과 상단 이동 링크 | 페이지의 마무리 정보 표현 |

시맨틱 태그는 코드 구조, 검색엔진 이해와 접근성에 도움이 됩니다. 반면 모든 영역에 의미 없는 태그를 억지로 적용하면 구조가 복잡해질 수 있으므로 스타일만 필요한 묶음에는 `div`를 사용했습니다.

### `<dialog>`와 직접 만든 모달

여행 생성·수정 창에는 브라우저가 제공하는 `<dialog>` 요소를 사용했습니다.

장점:

- 대화상자라는 의미가 HTML에 포함됩니다.
- `showModal()`과 `close()`로 열고 닫을 수 있습니다.
- 모달이 열린 동안 배경과 입력 흐름을 분리하기 쉽습니다.
- `::backdrop`을 이용해 배경을 스타일링할 수 있습니다.

단점:

- 오래된 브라우저에서는 지원 여부를 확인해야 합니다.
- 복잡한 포커스 정책이나 애니메이션은 추가 구현이 필요할 수 있습니다.

`div`로 직접 모달을 만들면 디자인과 동작을 자유롭게 제어할 수 있지만 `role="dialog"`, `aria-modal`, 포커스 이동, Escape 처리와 배경 입력 차단을 직접 구현해야 합니다. 이 프로젝트에서는 기본 의미와 동작을 활용할 수 있는 `<dialog>`가 적합하다고 판단했습니다.

### 하나의 상세 페이지와 URL 파라미터

도시마다 별도의 HTML을 만들지 않고 다음과 같은 하나의 주소 구조를 사용했습니다.

```text
destination.html?city=seoul
destination.html?city=busan
destination.html?city=jeju
```

JavaScript가 `URLSearchParams`로 `city` 값을 읽고 해당 도시 데이터를 렌더링합니다.

장점:

- 공통 레이아웃을 한 번만 관리할 수 있습니다.
- 새로운 도시를 추가할 때 데이터 중심으로 확장할 수 있습니다.
- 도시 페이지 간 디자인과 기능의 일관성을 유지할 수 있습니다.

단점:

- JavaScript가 실행되지 않으면 상세 콘텐츠가 표시되지 않습니다.
- 데이터 구조가 잘못되면 여러 도시 페이지에 동시에 영향을 줄 수 있습니다.
- 정적 HTML만 사용하는 방식보다 초기 구조를 이해하기 어려울 수 있습니다.

### ES Modules와 단일 JavaScript 파일

JavaScript를 한 파일에 모두 작성하지 않고 다음 역할로 분리했습니다.

```text
components/   영상 캐러셀, 오늘의 문구
core/         저장소, 인증, 공통 UI
data/         여행지 데이터
pages/        페이지별 동작
```

ES Modules는 필요한 기능을 `import`와 `export`로 연결해 파일의 책임과 의존 관계를 명확하게 만들 수 있습니다. 공통 인증과 저장소 코드를 여러 페이지에서 재사용할 수 있다는 장점도 있습니다. 반면 파일을 직접 열었을 때 브라우저 보안 정책으로 모듈이나 Fetch 요청이 제한될 수 있어 로컬 서버 실행이 필요합니다.

### Local Storage 선택

서버 없이도 회원과 여행 계획을 새로고침 후 유지하기 위해 Local Storage를 사용했습니다.

장점:

- 별도의 백엔드나 데이터베이스 없이 사용할 수 있습니다.
- JSON 문자열로 변환해 간단한 데이터를 저장하기 쉽습니다.
- 같은 브라우저에서는 새로고침 후에도 데이터가 유지됩니다.

단점:

- 브라우저와 기기가 달라지면 데이터가 공유되지 않습니다.
- 사용자가 직접 데이터를 수정하거나 삭제할 수 있습니다.
- 실제 서비스의 안전한 회원 인증과 권한 관리에는 적합하지 않습니다.
- 저장 용량이 제한적이며 복잡한 검색과 동기화가 어렵습니다.

따라서 이 프로젝트의 회원가입과 로그인은 프론트엔드 학습을 위한 실습용 인증이며 실제 서비스 인증을 대신하지 않습니다.

### Web Crypto API와 PBKDF2

비밀번호 원문을 그대로 저장하지 않기 위해 16바이트 임의 Salt와 PBKDF2 SHA-256을 사용해 파생된 값을 저장합니다. 반복 횟수는 `120,000`회로 설정했습니다.

원문 저장보다 직접적인 비밀번호 노출 가능성을 줄일 수 있지만, 처리 과정과 결과가 모두 사용자의 브라우저에 있고 데이터 저장소도 Local Storage이므로 실제 서버 인증과 동일한 보안 수준은 아닙니다. 이 구현은 비밀번호를 원문으로 다루지 않는 방식과 비동기 Web Crypto API를 학습하기 위한 선택입니다.

### Fetch API와 JSON

오늘의 여행 문구는 HTML이나 JavaScript에 직접 작성하지 않고 `quotes.json`에서 불러옵니다. 콘텐츠와 화면 로직을 분리할 수 있고 문구를 수정할 때 JavaScript를 변경하지 않아도 된다는 장점이 있습니다. 요청이 실패하거나 데이터가 비어 있을 때는 기본 문구를 표시하도록 예외 처리했습니다.

Fetch는 HTTP 환경을 기준으로 동작하므로 로컬 파일로 직접 실행하면 JSON 요청이 제한될 수 있습니다. 이 때문에 로컬 서버 실행을 기본 방법으로 안내합니다.

### 이벤트 위임

플래너의 여행 목록과 일정 목록은 JavaScript로 계속 생성·삭제됩니다. 각 버튼에 이벤트를 따로 등록하는 대신 부모 목록에서 클릭을 감지하고 `data-action`을 확인하는 이벤트 위임 방식을 사용했습니다.

동적으로 추가된 항목에도 별도의 이벤트 등록 없이 동작하고 이벤트 리스너 수를 줄일 수 있다는 장점이 있습니다. 다만 클릭된 요소에서 원하는 버튼을 찾는 과정과 `data-*` 규칙을 일관되게 관리해야 합니다.

### 접근성과 사용자 환경

| 적용 기술 | 적용 내용 |
| --- | --- |
| `aria-current="page"` | 현재 선택된 내비게이션 페이지 표시 |
| `aria-expanded` | 모바일 메뉴의 열림·닫힘 상태 제공 |
| `aria-live="polite"` | 달력 제목과 일정 목록 변경 안내 |
| `aria-label` | 아이콘 버튼과 동적 일정 버튼의 목적 설명 |
| `:focus-visible` | 키보드 사용자의 현재 포커스 표시 |
| `prefers-reduced-motion` | 모션 감소 설정 시 배경 전환과 애니메이션 최소화 |
| `loading="lazy"` | 화면 아래 명소 이미지의 초기 로딩 부담 감소 |

접근성 속성은 화면 디자인을 크게 바꾸지 않으면서 키보드 사용자와 보조 기술에 현재 상태와 버튼 목적을 전달합니다. 다만 속성만 추가한다고 접근성이 완성되는 것은 아니므로 실제 키보드 이동과 화면 읽기 흐름을 함께 확인해야 합니다.

## 구현 방식 요약

| 요구 사항 | 선택한 방식 | 대안 | 선택 이유 |
| --- | --- | --- | --- |
| 상단 메뉴 유지 | Sticky Header | Fixed Header | 문서 흐름을 유지하고 콘텐츠 겹침 감소 |
| 화면 고정 버튼 | Fixed | Sticky | 부모 영역과 관계없이 항상 같은 위치 유지 |
| 카드·달력 | CSS Grid | Flexbox | 여러 행과 열을 명확하게 제어 |
| 메뉴·버튼 정렬 | Flexbox | Grid | 한 방향 정렬을 간결하게 구현 |
| 여행 편집 창 | `<dialog>` | `div` 기반 모달 | 브라우저의 대화상자 의미와 기본 동작 활용 |
| 도시 상세 | URL 파라미터와 데이터 | 도시별 HTML | 중복 구조를 줄이고 데이터로 확장 |
| 코드 구조 | ES Modules | 단일 JS 파일 | 역할 분리와 공통 기능 재사용 |
| 데이터 유지 | Local Storage | 서버 DB | 백엔드 없이 브라우저에서 실습 가능 |
| 비밀번호 처리 | Web Crypto PBKDF2 | 원문 저장 | 원문 저장을 피하고 암호 처리 과정 학습 |
| 문구 데이터 | Fetch + JSON | HTML 하드코딩 | 콘텐츠와 화면 로직 분리 |
| 동적 목록 이벤트 | 이벤트 위임 | 개별 이벤트 등록 | 새로 생성된 항목도 동일하게 처리 |

## 데이터 저장 방식

회원 정보, 로그인 상태와 여행 계획은 브라우저 Local Storage에 저장됩니다. 서버에 저장되는 데이터가 아니므로 브라우저나 기기가 달라지면 기존 데이터가 공유되지 않습니다. 인증과 회원 목록 역시 프론트엔드 기술을 연습하기 위한 실습용 기능입니다.

## 실행 방법

ES Modules와 JSON 요청을 정상적으로 사용하려면 저장소 루트에서 로컬 서버를 실행합니다.

```bash
python3 -m http.server 8000
```

브라우저에서 아래 주소로 접속합니다.

```text
http://localhost:8000/assignment/day1/index.html
```

## 결과물 자기평가

여행지 소개에 그치지 않고 회원가입과 로그인, 사용자별 여행 계획 관리까지 하나의 흐름으로 연결했습니다. JavaScript를 역할별 ES Module로 분리하고 공통 저장소와 인증 기능을 재사용해 페이지가 늘어나도 관리할 수 있는 구조를 연습했습니다. 특히 달력 기반의 일정 관리와 준비 항목 CRUD를 직접 구현하면서 DOM 렌더링, 이벤트 처리와 상태 관리의 관계를 이해할 수 있었습니다.

Local Storage 기반이므로 여러 기기에서 데이터를 공유하거나 실제 사용자 권한을 안전하게 관리할 수 없다는 한계가 있습니다. 이후 백엔드나 데이터베이스와 연결한다면 실제 서비스에 가까운 인증과 동기화 기능으로 발전시킬 수 있습니다.
